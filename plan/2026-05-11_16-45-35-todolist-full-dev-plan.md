# Plan: ToDoList 完整开发计划

🎯 任务概述
当前仓库已经是一个可运行的 HarmonyOS Stage 示例工程，但现状只有静态任务列表和本地点击切换完成态，距离 `需求.txt` 里的完整目标还有明显差距。后续开发应以“保留现有单模块工程结构、逐步替换数据模型和页面结构”为主线，先打通核心待办闭环，再补提醒、动效和上架级体验。

本计划默认以 `entry` 单模块为实现边界，不额外拆分新模块；优先复用现有 `pages / view / viewmodel / common` 目录，只在必要时新增 `model / service / repository` 等最小目录。计划同时考虑并行执行，避免不同子任务改同一文件。

📋 执行计划
1. 建立现状基线与验收清单。
   先确认当前工程能正常构建、页面入口和资源引用无误，并把 `需求.txt` 转成可验收的功能清单。这里要明确当前基线只有静态列表、局部完成态、无新增任务、无本地持久化、无提醒。

2. 重构任务领域模型与排序规则。
   将 `Array<string>` 升级为 `Task` 实体，至少包含 `id`、`title`、`isCompleted`、`createdAt`、`dueAt`、`completedAt`、`reminderAt`。同步定义统一排序策略：已完成任务排前面并按完成时间倒序；未完成任务排后面并优先按到期时间升序、再按创建时间倒序，保证满足需求中的“完成后自动前置”。

3. 抽离状态管理与本地持久化。
   把当前 `DataModel` 从只读示例数据升级为真正的仓储层，负责初始化、增删改查、排序、序列化和落盘。持久化优先采用 HarmonyOS 当前工程可直接接入的本地键值存储方案，将 `Task[]` 序列化保存，确保重启应用后状态可恢复。

4. 重建主页面布局与信息架构。
   把 `Column + ForEach` 方案切换为更适合滚动列表的页面结构，补齐顶部欢迎语、当前日期、今日完成进度条、任务统计信息，并为列表区预留全屏滚动空间。此阶段同步设计空状态、长列表状态、即将到期和已逾期的视觉层级。

5. 实现任务录入、编辑与删除闭环。
   新增显眼的“添加任务”入口，弹出表单或底部面板，支持输入标题和可选截止时间。补齐任务删除、取消完成、编辑截止时间等基本操作，并保证新增后立即刷新列表、无页面重载。

6. 完成任务卡片与状态联动展示。
   将 `ToDoItem` 从局部 `@State` 改成由上层状态驱动，卡片上清晰展示标题、截止时间、完成时间、提醒状态和逾期标识。完成/取消完成时必须同步更新排序、完成时间和进度条，让视觉反馈与数据状态一致。

7. 实现到期提醒与通知能力。
   先完成前台视觉提醒逻辑：即将到期任务温和高亮，已逾期任务高亮警示。再评估当前课程环境可用的本地通知能力，补齐提醒注册、更新和取消逻辑；如果模拟器或权限环境限制后台通知，则先保证提醒调度接口完整，真机验证作为单独验收项。

8. 完成动画与视觉打磨。
   为新增、删除、完成状态切换、排序重排补上过渡动画，统一卡片圆角、阴影、间距和配色。这里要严格贴合需求中的“现代卡片风格”和“成熟商业应用质感”，但避免为了炫技引入重型动画框架。

9. 补齐测试与回归验证。
   以 Hypium 或当前工程可用测试方式补最小自动化测试，至少覆盖任务增删、状态切换、排序规则和持久化恢复。手工回归要覆盖冷启动恢复、长列表滚动、日期为空/已过期/即将到期、提醒开关切换、模拟器与真机差异。

10. 完成文档、截图与阶段提交。
    更新 README 的功能说明、运行方式和限制项，补充课程作业需要的截图或演示材料。按里程碑提交版本，确保每个阶段都可回退、可演示、可验收。

🔀 并行执行建议
- 第一轮并行：
  `worker-A` 负责任务模型、排序规则和仓储接口设计；
  `worker-B` 负责主页面信息架构与卡片样式草图；
  `explorer` 负责确认本地存储与通知 API 在当前 HarmonyOS 版本下的可用性。

- 第二轮并行：
  `worker-A` 落持久化和状态流；
  `worker-B` 落新增/编辑/删除交互；
  `worker-C` 落逾期样式、完成时间和进度条展示。

- 第三轮并行：
  `worker-A` 处理通知注册与取消；
  `worker-B` 处理动画和视觉细节；
  `explorer` 负责测试清单、回归脚本和 README 更新项。

✅ 阶段验收标准
- M1 核心闭环：可新增任务、可切换完成态、可自动排序、可显示完成时间。
- M2 数据可靠：应用重启后任务、完成状态、截止时间和完成时间均可恢复。
- M3 体验达标：长列表可顺畅滚动，逾期/临期视觉明确，顶部日期与进度信息正确。
- M4 提醒完成：至少前台提醒逻辑可用；若真机权限允许，本地通知可正常弹出。
- M5 作业交付：README、截图、回归记录齐全，仓库处于可演示状态。

⚠️ 风险与注意事项
- 当前工程是示例代码，`ToDoItem` 的完成态保存在组件内部，若不先上收状态，排序和持久化会反复返工。
- `需求.txt` 要求“已完成任务排前面”，这和常见待办应用习惯相反，开发和测试时必须以需求为准，不要擅自改成“未完成优先”。
- 本地通知能力可能受模拟器、权限或课程环境限制，必须提前做可行性验证，并准备“前台视觉提醒可交付、系统通知真机补验”的降级方案。
- 现有项目暂无测试目录，补测试时要控制范围，先覆盖纯逻辑层，不要一开始就做重 UI 自动化。
